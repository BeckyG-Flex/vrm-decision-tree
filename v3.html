Step 1 of 2 — Integration
How does this vendor connect to Flex?
Pick the scenario that best describes how this vendor works. Focus on the live setup, not just dev or test.
Scenario A — Flex calls out to the vendor
Your team uses the vendor's product or website. The vendor does not hold any credentials, login, or API key for a Flex system. Flex initiates all contact.
For example: a SaaS tool your team logs into, a service your engineers call from Flex code.
Scenario B — The vendor connects into Flex
The vendor holds credentials, an API key, or a token for a Flex system. Their platform can read from or write to Flex automatically, without someone manually triggering it each time.
For example: a payment processor that pushes transaction data into Flex's database, a monitoring tool with read access to Flex infrastructure.
Scenario C — Both directions, or the live setup is more complex
Data flows both ways, or the current dev/test setup looks like Scenario A but the live architecture will involve the vendor calling into Flex.
For example: a fintech partner where the integration spec is still being defined, or a vendor that starts as Scenario A but requires inbound access once a financial institution goes live.

Step 2 of 2 — Data Access
What data will this vendor have access to?
Select all types that apply. If you are not sure about a category, check it — Security will verify.
Highest sensitivity — any of these = T1
T1 Login credentials or access keys
The vendor stores or transmits passwords, PINs, or tokens that unlock a Flex system or a customer account.
For example: a password manager, an identity provider, a vendor that receives OAuth tokens for customer accounts.
T1 Government-issued identifiers
The vendor sees Social Security numbers, driver's license numbers, or passport numbers.
For example: a KYC/identity verification vendor, a credit bureau.
T1 Full bank account or card numbers
The vendor sees complete, unmasked financial account numbers — not last-4 or tokenized versions.
For example: a payment processor handling ACH transfers, a card issuer.
Sensitive data — T2
T2 Flex customer personal data
The vendor sees names, email addresses, phone numbers, or physical addresses belonging to Flex customers.
For example: a CRM, a customer communications platform, a fraud detection vendor.
T2 Flex employee personal data
The vendor sees names, email addresses, HR records, or any identifiable information about Flex employees.
For example: an HRIS, payroll provider, or benefits platform.
T2 Partial financial data
The vendor sees last-4 digits, masked account numbers, transaction amounts, or payment history — but not full account numbers.
For example: a reporting tool, a reconciliation platform, a customer-facing payment summary tool.
Standard — T3
T3 Business data only
Internal documents, roadmaps, metrics, or anonymized data. No customer or employee personal information.
For example: a project management tool, a design platform, an analytics dashboard with no PII.
